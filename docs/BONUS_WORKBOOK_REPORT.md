## 1. はじめに

### 1.1 背景

介護施設グループ「サフィール」では、毎年の賞与検討にあたり Excel 形式の賞与表を用いて、考課結果・順位・賞与金額・賞与引当金を管理してきました。しかし Excel ベースの運用には、次のような課題がありました。

- 施設ごとにファイルが分散し、本部による横断的な確認が難しい
- 施設長と本部で編集する項目が異なるが、権限の切り分けが曖昧になりやすい
- 過去年度のデータ参照がファイル探しに依存する
- 名簿変更や考課提出のたびに手作業での転記が発生する

本レポートでは、Web アプリケーション上に構築した**賞与表ワークスpeース**について、保持データ・保存設計・開発所感を整理します。

### 1.2 目的

賞与表ワークスペースは、次の目的で開発しました。

1. 考課システム（PostgreSQL）と連携し、名簿・採点データを自動反映する
2. 施設長と本部の役割に応じた編集権限を Web 上で実現する
3. 賞与金額・引当金を年度別に保存し、過去データも閲覧できるようにする
4. 引当金と確定賞与の差額をリアルタイムに可視化し、予算管理を支援する

## 2. システム概要

### 2.1 全体アーキテクチャ

賞与表ワークスペースは、React（フロントエンド）と FastAPI（バックエンド）で構成されています。画面表示時に複数のデータソースを統合し、ユーザー操作に応じて適切な保存先へ書き込みます。

```
┌─────────────────────────────────────────────────────────────────┐
│                    賞与表ワークスペース（React）                    │
│  ・施設選択 / 年度選択  ・表形式編集  ・サマリー表示  ・Excel出力   │
└────────────────────────────┬────────────────────────────────────┘
                             │ REST API
┌────────────────────────────▼────────────────────────────────────┐
│                      FastAPI（backend/app/main.py）               │
│  GET  /api/bonus-workbook          … 一覧取得・集計               │
│  PUT  /api/bonus-workbook          … ワークブック保存（手動）        │
│  PUT  /api/bonus-workbook/amounts  … 金額・引当金（自動保存）      │
│  POST /api/bonus-workbook/sync     … 考課データ反映               │
│  POST /api/bonus-workbook/sync-roster … 名簿反映                  │
└───────┬─────────────────┬──────────────────┬────────────────────┘
        │                 │                  │
        ▼                 ▼                  ▼
┌───────────────┐ ┌─────────────────┐ ┌──────────────────────────┐
│  PostgreSQL   │ │ JSON ファイル    │ │ Excel テンプレート         │
│  従業員・考課  │ │ bonus_amounts/  │ │ BONUS_WORKBOOK_TEMPLATE  │
└───────────────┘ └─────────────────┘ └──────────────────────────┘
```

### 2.2 画面の主な操作

| 操作 | 説明 | 保存タイミング |
|------|------|----------------|
| 名簿を反映 | DB の氏名・役職を賞与表行に同期 | 即時 |
| 考課から反映 | 提出済み考課の採点・特記・分析項目を自動入力 | 即時 |
| 保存 | 採点・順位・備考などワークブック項目を保存 | 手動（ボタン） |
| 金額入力 | 賞与金額（案/確定）・前年賞与を入力 | 自動（約0.7秒後） |
| 引当金設定 | 月額（千円）と月数（6/12）を設定 | 自動（約0.7秒後） |
| 年度切替 | 保存済み年度または現行年度を表示 | — |
| Excelダウンロード | テンプレート形式の Excel を出力 | — |

### 2.3 データ読み込みの流れ

1. PostgreSQL から対象施設の職員一覧を取得（Excel シートがない施設も名簿から行を生成）
2. Excel テンプレートにデータシートがあれば、行番号・レイアウト情報を参照
3. `backend/data/bonus_amounts/{施設}.json` から、選択年度の金額・引当金・保存済み項目をマージ
4. フロントエンドで表表示し、入力に応じてサマリー（合計・差額）をリアルタイム計算

## 3. 保持できるデータ

### 3.1 職員1行あたりの項目

1 職員につき 1 行で、以下のデータを保持・編集できます。**金額の単位はすべて千円**です。

#### （1）基本情報（名簿連携）

| 項目 | 内容 | 主な更新方法 |
|------|------|--------------|
| 社員ID | 例：i1001 | 名簿反映 |
| 氏名 | 職員名 | 名簿反映 |
| 役職 | リーダー、介護職員など | 名簿反映 |
| 所属施設 | 施設長ビュー時に表示 | 自動 |

#### （2）考課・評価関連

| 項目 | 内容 | 主な更新方法 |
|------|------|--------------|
| 自己評点 | 自己評価の合計点 | 考課反映 / 手入力 |
| 一次評点 | 第1評価者の合計点 | 考課反映 / 手入力 |
| 二次評点 | 第2評価者の合計点 | 考課反映 / 手入力 |
| 最終点 | 一次・二次の平均など | 自動計算 |
| 8点未満件数（自己） | 低評価項目の件数 | 考課反映 |
| 6点未満件数（他者） | 低評価項目の件数 | 考課反映 |
| カット項目 | 評価で問題となった項目 | 考課反映 |
| 昇格参考 | 昇格判断の参考情報 | 考課反映 |
| 昇給有 | ○ など | 考課反映 |
| 順位 | 考課点順の順位 | 考課反映（自動）/ 手入力 |
| ランク | A〜E（10%刻み等） | 考課反映（自動）/ 手入力 |
| 備考 | 特記事項（モーダル編集） | 手入力 |

#### （3）賞与金額関連

| 項目 | 内容 | 編集権限 |
|------|------|----------|
| 前年夏 賞与 | 前年度夏の賞与（千円） | 本部 |
| 前年冬 賞与 | 前年度冬の賞与（千円） | 本部 |
| 賞与金額（案） | 施設長案（千円） | 施設長 |
| 賞与金額 | 確定賞与（千円） | 本部 |

#### （4）施設単位の設定

| 項目 | 内容 | 編集権限 |
|------|------|----------|
| 引当金（月額） | 千円単位 | 施設長 |
| 引当金（月数） | 6ヶ月 or 12ヶ月 | 施設長 |
| 年間引当金合計 | 月額 × 月数 | 自動計算 |

### 3.2 集計データ（サマリーパネル）

画面下部で次の集計をリアルタイム表示します。

| 集計項目 | 計算方法 |
|----------|----------|
| 賞与金額（案）合計 | 全行の proposed_bonus_amount の合計 |
| 賞与金額合計 | 全行の bonus_amount の合計 |
| 社保込み合計 | 確定合計 ×（1 + 社保率 15%） |
| 賞与引当金（合計） | 月額 × 月数（施設長ビューは全施設合算） |
| 引当金との差額 | 引当金合計 − 社保込み合計（余剰/超過表示） |

### 3.3 年度別データ

賞与金額・引当金・ワークブック保存項目は**年度（fiscal_year）単位**で管理します。

- **現行考課期間の年度：** 権限に応じて編集可能
- **過去年度：** 施設長・本部とも閲覧のみ（`read_only` モード）

JSON ファイル内の構造例：

```json
{
  "amount_unit": "sen_yen",
  "years": {
    "2026": {
      "provision_monthly": 1100,
      "provision_months": 6,
      "rows": {
        "i1001": {
          "proposed_bonus_amount": 500,
          "bonus_amount": 450,
          "prior_summer_amount": 400,
          "prior_winter_amount": 380,
          "rank_order": 3,
          "rank_grade": "B",
          "note": "特記あり"
        }
      }
    }
  }
}
```

## 4. データの保存先と設計理由

### 4.1 保存先一覧

| # | 保存先 | パス / 場所 | 保存するデータ | 選定理由 |
|---|--------|-------------|----------------|----------|
| 1 | PostgreSQL | `employees`, `evaluations` 等 | 名簿・考課の本体データ | 人事考課システムの正規データ。他機能と共有 |
| 2 | JSON ファイル | `backend/data/bonus_amounts/{施設key}.json` | 賞与金額・引当金・順位・備考等（年度別） | 賞与表専用データ。年度をまたいで残し、Web から高速に読み書き |
| 3 | Excel テンプレート | `.env` の `BONUS_WORKBOOK_TEMPLATE_PATH` | 採点・順位等（シートがある施設） | 既存 Excel 運用との互換。ダウンロード用体裁の維持 |
| 4 | 施設設定 JSON | `backend/data/facilities.json` | 施設一覧・Excel シート名・引当金初期値 | 8 施設をコード変更なしで設定可能 |
| 5 | ブラウザ localStorage | `bonus-workbook-column-prefs:{施設}` | 列表示設定（カット/昇格参考） | UI 設定のみ。サーバー負荷不要 |

### 4.2 なぜ JSON に賞与データを保存したか

PostgreSQL の考課テーブルにすべてを入れる案もありましたが、次の理由から**賞与表専用の JSON ファイル**を採用しました。

1. **ライフサイクルの違い**  
   考課は「期間内に提出・確定」するが、賞与表は「施設長案 → 本部確定 → 引当金との突合」という別プロセスを持つ。

2. **年度をまたいだ参照**  
   過去の引当金・金額を見返す需要があり、年度キー付き JSON で履歴を残しやすい。

3. **段階的な導入**  
   全施設・全項目を DB テーブル化する前に、ファイルベースで素早く試作・運用開始できる。

4. **Excel との共存**  
   既存 Excel テンプレートを壊さず、Web 側のデータを上書きマージできる。

5. **権限のフィールド単位制御**  
   施設長が編集できる項目と本部が編集できる項目を、保存 API で細かく制御できる。

### 4.3 保存処理の2系統

#### 系統 A：金額・引当金（自動保存）

```
ユーザー入力
  → フロントエンド（700ms デバウンス）
  → PUT /api/bonus-workbook/amounts
  → bonus_amounts.py（save_amounts_for_workbook）
  → backend/data/bonus_amounts/{施設}.json
```

- 施設長：賞与金額（案）・引当金を保存
- 本部：確定賞与・前年夏/冬を保存
- 過去年度：API が 403 を返し保存不可

#### 系統 B：ワークブック本体（手動保存）

```
「保存」ボタン
  → PUT /api/bonus-workbook
  → bonus_workbook.py（Excel へ書込 + save_workbook_row_fields）
  → Excel テンプレート（シートがある場合）
  → backend/data/bonus_amounts/{施設}.json（WORKBOOK_ROW_FIELDS）
```

保存対象フィールド：`self_score`, `eval1_score`, `eval2_score`, `rank_order`, `rank_grade`, `note`

### 4.4 既存データの移行

初期実装では施設ごとに 1 ファイルのフラット構造でした。年度別対応に伴い、初回読込時に自動的に `years` 構造へ移行します。旧データが円単位（10,000 以上）の場合は千円へ変換するマイグレーションも実装しています。

## 5. 権限と運用フロー

### 5.1 ロール別権限

| ロール | 閲覧範囲 | 編集可能な項目（現行年度） |
|--------|----------|----------------------------|
| 施設長 | 自施設のみ | 賞与金額（案）、引当金、採点・順位・備考 |
| 本部・管理者 | 全施設 + 施設長ビュー | 確定賞与、前年夏/冬（引当金は閲覧のみ） |
| いずれも | 過去年度 | すべて閲覧のみ |

### 5.2 典型的な運用フロー

```
① 本部：考課期間を開始
② 職員・評価者：考課を提出
③ 施設長：「考課から反映」で採点・順位を取り込み
④ 施設長：賞与金額（案）と引当金を入力（自動保存）
⑤ 本部：各施設を確認し、確定賞与金額を入力
⑥ 本部：サマリーで引当金との差額を確認
⑦ 必要に応じて Excel ダウンロード
⑧ 翌年以降：年度プルダウンで過去データを参照
```

### 5.3 施設長ビュー（facility_directors）

本部のみアクセス可能な特殊ビューです。各施設の施設長を横断表示し、引当金は各施設 JSON から合算します。金額の編集は各施設の行に紐づく施設キー経由で保存されます。

## 7. 開発を通じて感じたこと

### 7.1 苦戦したこと

**（1）Excel と Web の二重管理**

いなは・そはらは Excel シートにデータ行がある一方、他施設は名簿（PostgreSQL）から行を生成するハイブリッド構成になりました。同じ UI で両方を扱うため、`bonus_workbook.py` に条件分岐が集中し、コードの見通しを保つのが難しかったです。

**（2）権限の細分化**

「案は施設長、確定は本部、引当金は施設長、過去は閲覧のみ」と、項目 × ロール × 年度の組み合わせが増えました。API 側の `allow_proposed` / `allow_bonus` / `allow_provision` と、フロント側の入力可否を常に一致させる必要があり、設計表を頭の中で整理しながら進めました。

**（3）単位の統一（千円）**

初期データや Excel 由来の値が円単位で混在していたため、1 万円以上の値は千円へ変換するマイグレーションを追加しました。現場の感覚（円）とシステム内部（千円）のズレは、仕様決定の早い段階で固定すべきだったと感じました。

**（4）UI のスクロール問題**

flex レイアウト内でテーブルが伸び、画面全体がスクロールしなくなる不具合が発生しました。`.bonus-workbook-scroll` でスクロール領域を切り分ける CSS 調整に時間を要しました。

**（5）年度別保存への移行**

フラットな JSON から `years` 構造への移行を、既存データを壊さず行う必要がありました。読込時マイグレーション方式を採用し、ユーザー操作なしで移行できるようにしました。

### 7.2 うまくいったこと

- 名簿取込と連動し、**Excel がなくても全施設で賞与表を開始**できるようになった
- 金額・引当金の**自動保存**により、保存忘れによるデータ消失リスクを低減できた
- **引当金と社保込み合計の差額表示**により、予算管理の判断がしやすくなった
- **年度プルダウン**で過去データを残しつつ、現行年度だけ編集可能にできた
- 施設長と本部の**役割分担が UI 上で明確**になった

### 7.3 もっとやりたかったこと

| 項目 | 内容 |
|------|------|
| DB テーブル化 | JSON から PostgreSQL へ移行し、同時編集・監査・バックアップを強化 |
| 完全スナップショット | 過去年度表示時に、その年の名簿構成を固定保存 |
| Excel 出力の充実 | ダウンロード Excel に金額列・引当金も反映 |
| 承認フロー | 施設長案 → 本部承認 → 確定ロックのワークフロー |
| 差額アラート | 引当金超過時の通知・警告表示 |
| 監査ログの可視化 | 誰がいつ金額を変更したかを画面で確認 |

## 9. まとめ

賞与表ワークスペースでは、**職員ごとの考課・順位・賞与金額・引当金を年度単位で保持**し、**PostgreSQL（名簿・考課）・JSON（賞与専用）・Excel（既存資産）** の3層にデータを分離して保存しました。これにより、施設長と本部の役割分担を Web 上で実現しつつ、過去データの参照も可能にしました。

開発を通じて、現場の Excel 文化と Web システムの両立、ならびに権限・年度・単位の設計が最も難しい要素であると実感しました。一方で、引当金と差額のリアルタイム表示など、Excel 単体では得にくい価値も確認できました。

## 付録 A：提出課題 3 問への直接回答

### Q1. どんなデータを保持できるようにしたか

職員ごとに、考課点・順位・ランク・備考・前年賞与（夏/冬）・賞与金額（案/確定）を保持できる。施設ごとに賞与引当金（月額×月数）を設定でき、社保込み合計と引当金の差額を自動集計する。すべて年度単位で管理し、過去年度は閲覧可能。

### Q2. どこに、なぜ保存したか

- **PostgreSQL** … 名簿・考課の正データ（他機能と共有するため）
- **`backend/data/bonus_amounts/*.json`** … 賞与表専用の年度別データ（Web からの高速読み書き・履歴参照のため）
- **Excel テンプレート** … 既存フォーマットとの互換・ダウンロード用
- **`facilities.json`** … 施設・シート名・引当金初期値の設定

考課システムと賞与表の更新タイミングが異なり、年度をまたいで残したいデータがあるため、本体 DB と賞与表データを分離した。

### Q3. 作ってみてどう感じたか

Excel 文化の現場を Web 化するには、既存資産との共存と施設長/本部の役割分担設計が最も難しかった。権限・年度・単位（千円）の整理に時間を要した。一方、引当金と差額のリアルタイム表示、金額の自動保存、全施設対応など、運用上のメリットを実感した。今後は DB 化・承認フロー・完全スナップショットなどに広げたい。
